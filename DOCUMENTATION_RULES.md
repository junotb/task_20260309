# XML 문서 작성 규칙

## 1. 엔티티 및 DTO: "데이터의 성격과 예시"

- 속성(Property)에는 **단순히 이름만** 적지 않음
- **비즈니스 제약 조건** 또는 **Swagger 테스트용 예시값** 반드시 포함
- 무의미한 주석 지양: `/// Id입니다.` → 금지. 차라리 비워둠

## 2. 인터페이스: "구현자가 지켜야 할 계약(Contract)"

- 이 기능을 **사용하는 사람** + **구현할 후임자**를 위한 가이드
- 구현 시 반드시 준수해야 할 동작, 예외, 성능, null 처리 등 명시

## 3. 비즈니스 로직(Handler/Service): "엣지 케이스와 부수 효과"

- 코드 흐름은 코드가 설명
- **"이 로직이 실패하면 어떻게 되는가?"**
- **"비동기로 어떤 일이 일어나는가?"**
- 이런 내용은 주석으로 명시

## 4. API 엔드포인트: "클라이언트를 위한 응답 사전"

- Swagger 핵심 구간
- 성공(200, 201, 202)뿐 아니라 **실패(400, 404)가 발생하는 구체적인 이유** 기술
